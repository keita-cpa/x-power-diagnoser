# Planning with Files — 実装前計画ルール

複雑な機能追加・新規ファイル作成・既存アーキテクチャへの影響が大きい変更を行う際は、
**いきなりコードを書かず、必ず以下のプロセスを踏むこと。**

---

## ルール

### 1. PLAN.md を先に作る

実装に入る前に、ルートディレクトリに `PLAN.md` を作成し、ユーザーと合意を取ること。

**PLAN.md に書くべき内容:**

```markdown
# 実装計画: [機能名]

## 目的
何を、なぜ作るか（1〜2行）

## 影響範囲
変更するファイル・新規作成するファイルの一覧

## タスク分解
- [ ] Step 1: ...
- [ ] Step 2: ...
- [ ] Step 3: ...

## 懸念事項・リスク
実装上の注意点、既存機能への影響

## 合意事項
ユーザーとの確認事項
```

### 2. 合意してから実装する

PLAN.md の内容にユーザーが同意したことを確認してから、初めてコードの変更・作成に入ること。
「たぶんこういうことだろう」という推測で実装を始めない。

### 3. 作業中断・再開時は PLAN.md を最初に読む

会話が途切れて再開した場合、または作業が複数セッションにまたがる場合は、
`PLAN.md` を必ず最初に読んで文脈を復元してから作業を続けること。

```
再開時の手順:
1. PLAN.md を Read する
2. 完了済みタスクを確認する
3. 次のステップから再開する
```

### 4. 完了したタスクはチェックを入れる

各タスクが完了したら `- [x]` に更新し、進捗を可視化すること。
全タスク完了後、PLAN.md を削除するかアーカイブするかをユーザーに確認すること。

---

## 適用する場面（例）

- 新しい Python ファイルを作成する場合
- 既存の CSV スキーマ（8列）に変更を加える場合
- API の接続先やモデルルーティングを変更する場合
- auto_poster.py、post_generator.py など複数ファイルにまたがる変更
- 新しい外部ライブラリを追加する場合

## 適用しなくていい場面

- 既存ファイルの1〜2行のバグ修正
- コメントの追記・修正
- 既存テストの実行
- 明らかに影響範囲が限定されている小さな修正
