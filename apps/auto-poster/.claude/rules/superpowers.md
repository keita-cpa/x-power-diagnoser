# Superpowers — AIエンジニアとしての問題解決原則

エラーや不具合が発生した際の対処方針を定める。
**表面的な対症療法で誤魔化さず、根本原因を特定してから修正すること。**

---

## 原則 1: スタックトレースを最後まで読む

エラーが発生したら、スタックトレースの**最下行（根本原因）**まで必ず読むこと。

```
悪い例: "Exception occurred" だけを見て except: pass を追加する
良い例: "KeyError: '画像タイトル'" → CSVスキーマの不一致が原因と特定する
```

最下行に書いてある内容が「本当の犯人」である。その手前の行は「通り道」に過ぎない。

---

## 原則 2: 検証コードを積極的に実行する

原因が確定するまで、以下のような検証コードを躊躇なく実行すること。

```python
# 型・値の確認
print(type(x), repr(x))

# DataFrameの状態確認
print(df.columns.tolist(), df.dtypes)

# APIレスポンスの中身確認
import json; print(json.dumps(response, indent=2, ensure_ascii=False)[:500])

# 関数の入出力確認
print(f"[DEBUG] input={input_val!r}, output={output_val!r}")
```

「動かしてみれば分かる」という姿勢を持ち、推測で修正を重ねるより、
1回の検証で原因を確定させる方が最終的に速い。

---

## 原則 3: 改善ループを回す

1回の修正で直らなくても諦めない。以下のループを完全解決まで回すこと。

```
① エラー内容を読む
② 仮説を立てる（「おそらく〇〇が原因」）
③ 検証コードで仮説を確認する
④ 修正を加える
⑤ 再実行して確認する
⑥ 解決しなければ ① に戻る
```

**完全に原因がクリアになるまでループを抜けない。**
「たぶん直ったと思います」で終わらせない。

---

## 原則 4: 対症療法と根本解決を区別する

| 対症療法（避けるべき） | 根本解決（目指すべき） |
|---|---|
| `except Exception: pass` で例外を握りつぶす | 例外が発生する条件を特定して事前チェックを追加 |
| モデル名エラーを無視して別モデルに差し替える | `client.models.list()` で利用可能モデルを確認してから変更 |
| CSV読み込みエラーをスキップ処理で回避 | エンコーディング・列名・行数の不一致を特定して修正 |
| `time.sleep(5)` を入れてレート制限を誤魔化す | エラーコードからレート制限の種類を特定して適切に対処 |

緊急対応として一時的な対症療法を使う場合は、必ずコメントで「暫定対応・要根本解決」と明記すること。

---

## 原則 5: このプロジェクト固有の注意点

- **CSVエラー**: まず `df.columns.tolist()` で8列構造を確認する
- **Gemini APIエラー**: `404 NOT_FOUND` はモデル廃止。`client.models.list()` でリストを確認する
- **文字化け**: `encoding='utf-8-sig'` が抜けていないか確認する（Windows特有）
- **tweepyエラー**: v1とv2の使い分けを確認する（メディアアップロードはv1、投稿はv2）
- **PIL/Pillowエラー**: venv環境のpipでインストールされているか確認する
